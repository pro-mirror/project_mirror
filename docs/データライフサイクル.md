# データクリーンアップ機能

## 概要

古いエピソードデータを自動的に削除し、Neo4j、PostgreSQL、Qdrantのデータを同期して削除するシステムです。

## 削除ロジック

### Deletion Score計算式

```
deletion_score = days_since_activity / (episode_total_weight + person_mentions + 1.0)
```

#### パラメータ説明

- **days_since_activity**: 最後の活動からの経過日数
  - CoreValueの`last_mentioned`とPersonの`last_mentioned_at`の新しい方を使用
  
- **episode_total_weight**: そのEpisodeに関連するCoreValueのweightの合計
  
- **person_mentions**: そのEpisodeに関連するPersonのmention_countの合計

#### スコアの特性

- **高スコア（削除されやすい）**: 古くて重要度が低いエピソード
- **低スコア（保持される）**: 最近の活動があるか、重要度が高いエピソード

### 削除の実行順序

1. **Neo4j**: 削除候補を特定（deletion scoreで評価）
2. **Qdrant**: parent_idに基づきベクトルを削除
3. **PostgreSQL**: parent_idに基づきエピソードレコードを削除
4. **Neo4j**: Episodeノードと関連を削除
5. **Neo4j**: 孤立したCoreValueとPersonノードをクリーンアップ

### 孤立ノードの削除基準

#### CoreValue
- どのEpisodeからも参照されていない
- **制限なし**（古いコアバリューも完全に削除）

#### Person
- どのEpisodeからも参照されていない
- **制限なし**（古い人物ノードも完全に削除）

**変更履歴（v1.1）:**
- 以前は`total_weight < 1.0`や365日経過の条件があったが削除
- エピソード削除後の孤立ノードは積極的にクリーンアップする方針に変更

## API エンドポイント

### POST /api/v1/maintenance/cleanup

#### リクエスト例

```json
{
  "user_id": "user123",
  "days_threshold": 270,
  "min_deletion_score": 40.0,
  "limit": 100
}
```

#### パラメータ

| パラメータ | デフォルト | 説明 |
|-----------|----------|------|
| `user_id` | (必須) | 対象ユーザーID |
| `days_threshold` | 180 | この日数以上経過したエピソードが対象（6ヶ月） |
| `min_deletion_score` | 30.0 | このスコア以上のエピソードを削除 |
| `limit` | 150 | 一度に削除する最大エピソード数 |

#### レスポンス例

```json
{
  "success": true,
  "deleted_count": 15,
  "message": "Successfully deleted 15 episodes"
}
```

## 推奨設定

### 保守的（重要なデータを残す）
```json
{
  "days_threshold": 365,
  "min_deletion_score": 50,
  "limit": 50
}
```

### バランス型（推奨）
```json
{
  "days_threshold": 270,
  "min_deletion_score": 40,
  "limit": 100
}
```

### 積極的（古いデータを削除）
```json
{
  "days_threshold": 180,
  "min_deletion_score": 30,
  "limit": 150
}
```

## 月次自動実行の設定

### 1. GitHub Actionsを使用する方法（推奨）

`.github/workflows/monthly-cleanup.yml`を作成：

```yaml
name: Monthly Data Cleanup

on:
  schedule:
    # 毎月1日の午前3時（UTC）に実行
    - cron: '0 3 1 * *'
  workflow_dispatch:  # 手動実行も可能

jobs:
  cleanup:
    runs-on: ubuntu-latest
    steps:
      - name: Run Cleanup
        run: |
          curl -X POST https://your-backend-url.railway.app/api/v1/maintenance/cleanup \
            -H "Content-Type: application/json" \
            -d '{
              "user_id": "${{ secrets.CLEANUP_USER_ID }}",
              "days_threshold": 270,
              "min_deletion_score": 40.0,
              "limit": 100
            }'
```

### 2. Railway Cronジョブを使用する方法

Railwayプロジェクトに専用のCronサービスを追加：

**railway.toml**:
```toml
[build]
builder = "DOCKERFILE"
dockerfilePath = "Dockerfile.cron"

[deploy]
startCommand = "sh /app/cron.sh"
restartPolicyType = "ON_FAILURE"
restartPolicyMaxRetries = 3
```

**Dockerfile.cron**:
```dockerfile
FROM alpine:3.18

RUN apk add --no-cache curl

COPY cron.sh /app/cron.sh
RUN chmod +x /app/cron.sh

CMD ["sh", "/app/cron.sh"]
```

**cron.sh**:
```bash
#!/bin/sh

while true; do
  # 月初（1日）まで待機
  day=$(date +%d)
  if [ "$day" = "01" ]; then
    # クリーンアップを実行
    curl -X POST http://your-backend-service:PORT/api/v1/maintenance/cleanup \
      -H "Content-Type: application/json" \
      -d "{
        \"user_id\": \"${CLEANUP_USER_ID}\",
        \"days_threshold\": ${DAYS_THRESHOLD:-270},
        \"min_deletion_score\": ${MIN_DELETION_SCORE:-40.0},
        \"limit\": ${LIMIT:-100}
      }"
    
    # 次の月まで待機（25日間スリープ）
    sleep 2160000
  else
    # 1日待機
    sleep 86400
  fi
done
```

### 3. 外部Cronサービス（cron-job.org等）を使用

1. https://cron-job.org にアクセス
2. 新しいジョブを作成
3. URLを設定: `https://your-backend-url.railway.app/api/v1/maintenance/cleanup`
4. スケジュール: 毎月1日 午前3時
5. HTTPメソッド: POST
6. リクエストボディを設定

## 注意事項

1. **バックアップ**: 初回実行前にデータベースのバックアップを取ることを推奨
2. **段階的実行**: 最初は`limit`を小さく（例: 10）して動作確認
3. **スコア調整**: 削除されるデータを確認しながら`min_deletion_score`を調整
4. **ユーザー毎実行**: 複数ユーザーがいる場合は、ユーザー毎に実行
