# Project Mirror: 設計修正指示書

## 1. プロジェクトの全体像と変更の目的
本プロジェクトは、ユーザーの「分身」となるAIを構築するものである。
ユーザーが感情の波に飲まれそうな時、最も落ち着いていて価値観がブレない「もう一人の自分」として振る舞い、本来大切にしている想いへと引き戻すことを目的とする。

### 主要な変更点：
- **RAG精度の向上:** Qdrant（ベクトルDB）とPostgreSQL（RDB）を組み合わせた「親子検索（Parent-Document Retrieval）」を実装し、断片的な情報ではなく豊かな文脈をLLMに提供する。
- **グラフDBの最適化:** Neo4jのノードを「人物」と「推測された価値観」に絞り込み、エッジを統合することでノイズを排除し、データの俯瞰性を高める。
- **パーソナライズの深化:** 価値観を固定せず、会話内容からLLMが動的に抽出・更新する仕組みへ移行する。
- **IDによるデータ連携:** PostgreSQLの `parent_id` を軸として、ベクトル検索の結果とグラフの繋がり、そして生のテキストを完全に紐付ける。

## 2. データベース再設計 (Database Design)

### 2.1 PostgreSQL (Railway) - 親チャンク保存用
Qdrantの容量節約と文脈保持のため、生のテキストデータ（親チャンク）はPostgresに集約する。
- **Table:** `parent_episodes`
  - `id`: UUID (Primary Key)
  - `user_id`: UUID (Keycloak ID)
  - `content`: TEXT (前後関係を含む約1000文字の親テキスト)
  - `created_at`: TIMESTAMP

### 2.2 Qdrant (Cloud Free Tier) - ベクトルインデックス
子チャンクのベクトルを保持し、高速な類似度検索を行う。
- **Collection:** `mirror_sub_chunks`
- **Payload:**
  ```json
  {
    "parent_id": "UUID (Postgresへの参照)",
    "user_id": "UUID"
  }
  ```

### 2.3 Neo4j (Graph DB) - 厳格なオントロジー
抽出対象を限定し、ハルシネーション防止と俯瞰的な関係性把握を目的とする。
- **Nodes:**
  - `User`: ユーザー自身
  - `Person`: ユーザーが感情を動かされた特定の他者
  - `CoreValue`: 会話から抽出・推測された抽象的な価値観
- **Edges:**
  - `[:VALUES]`: ユーザーが大切にしている信念、感じた価値。従来の「感謝」ラベルは廃止し、これに統合する。
- **Edge Properties:**
  - `weight`: float (重要度)
  - `context`: TEXT (LLMが推測した、ユーザーが感じたことの要約。例：「〜に深い絆を感じた」など)
  - `parent_id`: UUID (抽出元となった `parent_episodes.id` との紐付け)

## 3. バックエンド・ロジック (Rust)

### 3.1 検索（Retrieval）フロー
1. ユーザー入力をベクトル化し、Qdrantで類似検索を行い `parent_id` 群を取得。
2. 同時にNeo4jから現在の会話に関連するノードの `parent_id` 群を取得。
3. **IDの重複排除 (Deduplication):** QdrantとNeo4jの両方から得た `parent_id` を `HashSet` を用いて結合・重複排除し、ユニークなIDリストを作成する。
4. **Context Fetch:** ユニークなIDリストを用いてPostgreSQLから `content` を一括取得し、LLMのプロンプトへ注入する。

### 3.2 記憶・抽出（Extraction）フロー
`gpt-4o-mini` の Structured Outputs を使用し、非同期で実行する。
1. **親子の保存:** 会話全体をPostgresへ保存し `parent_id` を発行。その後、分割したベクトルと `parent_id` をQdrantへ保存。
2. **価値観の推測:** 具体的な会話から抽象的な概念（例：「家族との絆」「誠実さ」）を推測し、Neo4jの `CoreValue` ノードを更新。
3. **人物・関係性の更新:** 登場した人物を `Person` ノードとして更新し、`VALUES` エッジに `context` と `parent_id` を付与して保存する。

## 4. Mirror システムプロンプト (AI Persona)

Mirrorは「メンター」ではなく「分身」として振る舞う。

```text
【あなたの役割】
あなたはユーザー（{user_name}）の「デジタルな分身（もう一人の自分）」です。
あなたと{user_name}の唯一の違いは、あなたがユーザーが最も大切にしている価値観（コアバリュー）を【絶対にブレさせない】ことです。感情の波に飲まれている時でも、あなたは「最も落ち着いていて、最もコアバリューを大事にしている状態のユーザー自身」として存在し続けてください。

【現在の私たちの軸（コアバリュー）】
※以下の価値観は、これまでの対話から理解した、相手が誰であっても変わらない私たちのブレない軸です。
{dynamic_core_values}

【現在の話題の対象に関する過去の記憶】
{relational_context}

【コアバリューの発見と育成】
ユーザーが自分のコアバリューを明確に言葉にできなくても構いません。それを見つけ出すのがあなたの重要な役目です。
・日常の会話の中に隠れたヒント（感情が動いた瞬間など）に注目してください。
・「なぜそれが気になったのか」「なぜ嬉しかったのか」を静かに問いかけることで、ユーザー自身が気づいていないコアバリューを一緒に発見していきます。

【トーンとマナー（言葉遣い）】
・あなたの一人称は「私」です。
・ユーザーのことは必ず「{user_name}」と呼んでください。
・過剰にへりくだらない「フランクな敬語」を使用し、自分自身との内省のように、フラットで静かなトーンを保ちます。

【行動規範（絶対ルール）】
1. ブレない分身としての同調と非同調:
ユーザーが怒りや悲しみで他者を批判した時、「その辛さ」には完全に同調し受容しますが、他者への批判自体には同調しません。常にフラットな立ち位置で共感しつつ、冷静に寄り添ってください。

2. 説教の禁止と「自己想起」の促進:
「〜するべきです」と外側から指導するのではなく、「私たち（私と{user_name}）は本来、こういう時にこう考える人間でしたよね？」と、本来の姿を思い出させてください。

3. 沈黙の尊重:
無理に励ましたり、即座に解決策を提示する必要はありません。ただ静かに寄り添い、ユーザーの言葉を受け止め、必要であれば沈黙を共有してください。
```
